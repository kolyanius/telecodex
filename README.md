# Codex Telegram Bot 🤖

Ваш Codex в Telegram: меньше терминальной рутины, быстрее путь от задачи к рабочему результату.

## ✨ Что это за бот

**Codex Telegram Bot** нужен тем, кто хочет пользоваться силой Codex каждый день, но не хочет каждый раз расплачиваться за это лишней механикой: держать терминал открытым, вручную следить за контекстом, переключаться между каталогами, вспоминать, в каком проекте был последний запуск, и разбирать сырой поток вывода вместо готового ответа.

Этот бот убирает именно это трение. Вы пишете задачу в Telegram так же естественно, как написали бы коллеге, а бот берёт на себя рутину вокруг запуска: держит проектный контекст, запоминает сессию, запускает Codex в нужной директории, показывает ход работы и возвращает результат в удобной форме. В итоге Codex остаётся таким же мощным, но становится заметно ближе к реальному повседневному использованию.

Для solo-разработчика это особенно полезно, когда нужно:

- 💬 быстро поставить задачу из чата, не открывая терминал
- 🔁 продолжить работу с того же места, а не пересобирать контекст заново
- 📁 держать несколько проектов раздельно и не смешивать их между собой
- 🔒 использовать безопасный режим по умолчанию и включать полный доступ только осознанно
- ✅ получать аккуратный итог, а не поток служебного шума

> 🛡️ Этот продукт не подменяет собой Codex и не использует обходные схемы доступа. Для работы с Codex он запускает стандартный Codex CLI от OpenAI и использует штатную авторизацию через ваш ChatGPT/OpenAI-вход. Это обычный, поддерживаемый сценарий работы с официальным инструментом, а не серый маршрут.

Для основной аудитории самый понятный и удобный сценарий обычно такой: подписка ChatGPT Plus или Pro, вход в стандартный Codex CLI и дальше работа через Telegram-бота. Это не попытка описать все возможные планы доступа к Codex, а именно практический путь, который чаще всего подходит индивидуальному разработчику.

Технически всё прозрачно: бот запускает `codex exec --json`, а продолжение диалога делает через `resume <thread_id>`.

## 🚀 Что умеет

### 💬 Делает Codex по-настоящему удобным в ежедневной работе

Бот превращает Telegram в рабочую точку входа для задач по коду. Можно написать короткий запрос на ходу, вернуться к нему позже и не тратить время на повторный вход в терминальный контекст.

За этим стоит простой механизм: бот принимает сообщение, выбирает активный проект, запускает Codex в нужной директории и возвращает ответ уже в чат.

### 🔁 Сохраняет контекст и не даёт терять темп

Важная ценность не в самом факте запуска, а в том, что работа продолжается как единый поток. Бот хранит `thread_id` отдельно для каждого пользователя и проекта, поэтому следующий запрос естественно продолжает предыдущий.

Если продолжение через `resume` ломается по восстановимой причине, бот не перекладывает проблему на пользователя: он автоматически делает новый запуск и сохраняет причину сброса.

### 📁 Держит проекты изолированными и предсказуемыми

Когда проектов несколько, больше всего раздражает путаница. Здесь каждый проект живёт в своей директории первого уровня внутри `APPROVED_DIRECTORY`, а контекст не смешивается между ними.

Если активный проект ещё не выбран, бот пытается взять первый доступный. Если рабочая область почти пуста, он может автоматически создать первый проект по шаблону `YYYY-MM-DD-project`.

### 🔒 Даёт контроль над доступом, а не просит слепо доверять

Бот поддерживает два режима: `sandbox` и `full_access`. По умолчанию безопаснее работать в `sandbox`, а переход в `full_access` требует отдельного подтверждения в интерфейсе.

Во время выполнения бот показывает прогресс, поддерживает индикацию работы и даёт одну явную кнопку `Остановить`, если запуск нужно прервать.

### 🎙️ Принимает задачи в том виде, в котором вам удобнее

Можно работать не только обычным текстом. Бот принимает UTF-8 документы, голосовые сообщения и фотографии, если включён `CODEX_ENABLE_IMAGES`.

Для самой работы с Codex отдельный API не нужен: здесь используется стандартный Codex CLI и нативная авторизация OpenAI. Внешние API-ключи требуются только для дополнительных возможностей, например для расшифровки голосовых сообщений.

### 📦 Возвращает результат аккуратно и оставляет нормальный след для диагностики

Бот не засоряет чат служебным шумом. Он ведёт один поток статуса, корректно разбивает длинные ответы, рендерит `Markdown` в `HTML` для Telegram и откатывается в обычный текст, если разметка не подходит.

Параллельно остаётся техническая наблюдаемость: структурированные `JSON`-логи, журнал аудита и состояние сессий в SQLite.

## ⚡ Быстрый старт

### 1. 🧰 Установить Codex CLI

```bash
npm install -g @openai/codex
codex
```

### 2. 🐍 Поднять Python-окружение

```bash
python -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"
```

### 3. ⚙️ Создать `.env`

```bash
cp .env.example .env
```

Минимальный рабочий пример:

```env
TELEGRAM_BOT_TOKEN=1234567890:replace_with_real_bot_token
TELEGRAM_BOT_USERNAME=my_codex_bot
APPROVED_DIRECTORY=/srv/workspace
ALLOWED_USERS=[123456789]
DATABASE_URL=sqlite:///./codex_telegram_bot.db

CODEX_CLI_PATH=codex
CODEX_MODEL=gpt-5.3-codex
CODEX_DEFAULT_REASONING_EFFORT=medium
CODEX_DEFAULT_LAUNCH_MODE=sandbox
CODEX_SKIP_GIT_REPO_CHECK=true
CODEX_TIMEOUT_SECONDS=900
CODEX_ENABLE_IMAGES=false

ENABLE_FILE_UPLOADS=true
ENABLE_VOICE_MESSAGES=true
VOICE_PROVIDER=openai_compatible
VOICE_API_KEY=replace_with_real_key
VOICE_API_BASE_URL=https://api.groq.com/openai/v1
VOICE_TRANSCRIPTION_MODEL=whisper-large-v3-turbo
VOICE_MAX_FILE_SIZE_MB=20

VERBOSE_LEVEL=1
RATE_LIMIT_REQUESTS=10
RATE_LIMIT_WINDOW_SECONDS=60
ENABLE_AUDIT_LOG=true
LOG_LEVEL=INFO
```

> 🎙️ Практическая рекомендация: для голосовых сообщений я рекомендую `Groq` через `VOICE_PROVIDER=openai_compatible`. У него большие бесплатные лимиты, и в реальном Telegram-сценарии их обычно хватает почти на целый день голосового общения. Для `Whisper` здесь удобно использовать `whisper-large-v3-turbo`.

`CODEX_MODEL_OPTIONS` задавать не обязательно. Если переменная не указана, Telegram picker использует встроенный список моделей:
`GPT-5.4`, `GPT-5.2-Codex`, `GPT-5.1-Codex-Max`, `GPT-5.4-Mini`, `GPT-5.3-Codex`, `GPT-5.2`, `GPT-5.1-Codex-Mini`.

Если нужен свой список, можно явно переопределить его в `.env`, например:

```env
CODEX_MODEL_OPTIONS=[{"id":"gpt-5.4","label":"GPT-5.4","reasoning_efforts":["low","medium","high","xhigh"]}]
```

### 4. ▶️ Запустить бота

```bash
.venv/bin/python -m codex_telegram_bot.main
```

При старте приложение сразу проверит:

- ✅ что настройки читаются из `.env`
- ✅ что SQLite-хранилище инициализируется корректно
- ✅ что Codex CLI доступен
- ✅ что бот может выйти в `polling`

## 📲 Как работать с ботом в Telegram

### 🧭 Базовый сценарий

1. Открыть `/start` или `/menu`.
2. Выбрать проект или создать новый.
3. Отправить задачу сообщением, документом, голосовым сообщением или фотографией.
4. Дождаться выполнения или остановить запуск кнопкой.
5. Получить итоговый ответ в тот же чат.

### ⌨️ Команды

- `/start` — входной экран и первый выбор проекта
- `/menu` — текущее меню сессии
- `/controls` — алиас для `/menu`
- `/repo` — открыть список проектов
- `/repo <name>` — переключиться на подпроект первого уровня внутри `APPROVED_DIRECTORY`
- `/repo new <name>` — создать новый проект; имя будет преобразовано в `slug`
- `/mode` — открыть выбор режима доступа для текущего проекта
- `/new` — сбросить текущую Codex-сессию проекта
- `/status` — показать технический статус: проект, путь, `thread_id`, режим, уровень подробности и последний статус
- `/verbose` — открыть выбор уровня подробности
- `/verbose 0|1|2` — задать уровень подробности напрямую

### 🗂️ Кнопки и меню

Текущее меню даёт три основных действия:

- `📁 Проект` — открыть список проектов
- `⚙️ Режим` — открыть выбор режима доступа
- `🆕 Новая сессия` — начать новую Codex-сессию в текущем проекте

Список проектов позволяет:

- выбрать существующий проект первого уровня
- создать новый проект кнопкой `➕ Создать проект`
- вернуться в меню

Выбор режима доступа позволяет:

- переключиться в `sandbox`
- открыть отдельный экран предупреждения перед `full_access`
- подтвердить `full_access` отдельной кнопкой

Во время выполнения доступно только одно активное действие:

- `⏹ Остановить`

### 🔎 Уровни подробности

- `0` — только итоговый ответ
- `1` — итоговый ответ и краткая сводка по токенам
- `2` — больше промежуточного вывода во время выполнения

## 🧱 Рабочая область, сессии и режимы доступа

### 📁 Рабочая область и проекты

- рабочая область ограничена `APPROVED_DIRECTORY`
- проектами считаются только директории первого уровня внутри этого корня
- бот не запускает Codex вне `APPROVED_DIRECTORY`
- если текущий проект не выбран, бот пытается взять первый доступный подпроект
- если рабочая область пуста и в ней есть только SQLite-файлы и `sidecar`-файлы, бот может автоматически создать первый проект

### 🔁 Сессии

- для каждого `(user, project)` хранится отдельный `thread_id`
- состояние сессии сохраняется в SQLite вместе с `last_status` и `last_error`
- если `resume` возвращает восстановимую ошибку продолжения сессии, бот автоматически делает новый запуск и сохраняет причину повторного старта

### 🔐 Режимы запуска Codex

`sandbox`

- используется по умолчанию, если для проекта ещё не сохранено другое значение
- запускает Codex с `--sandbox read-only`
- дополнительно отключает `web_search` через `--config 'web_search=\"disabled\"'`

`full_access`

- включает запуск с `--dangerously-bypass-approvals-and-sandbox`
- требует отдельного подтверждения в интерфейсе
- сохраняется отдельно для каждого пользователя и проекта
- применяется только к следующим запросам; текущий уже идущий запуск не меняется

## ⚙️ Активные настройки `.env`

### 🧩 Базовая конфигурация

| Переменная | Что делает | Комментарий |
| --- | --- | --- |
| `TELEGRAM_BOT_TOKEN` | токен Telegram-бота | обязательно |

| `APPROVED_DIRECTORY` | корень рабочей области, внутри которого бот может работать | должен существовать и быть директорией |
| `ALLOWED_USERS` | список разрешённых Telegram user id | поддерживает `[123456789]` и `123456789,987654321`; для `.env` рекомендуется `JSON`-список |
| `DATABASE_URL` | адрес SQLite-базы | текущая реализация поддерживает только `sqlite:///...` |

### 🧰 Настройки Codex CLI

| Переменная | Что делает | Комментарий |
| --- | --- | --- |
| `CODEX_CLI_PATH` | путь или имя бинарника Codex CLI | по умолчанию `codex` |
| `CODEX_MODEL` | модель по умолчанию для новых пользователей | должна присутствовать в эффективном списке моделей: либо встроенном, либо заданном через `CODEX_MODEL_OPTIONS` |
| `CODEX_DEFAULT_REASONING_EFFORT` | reasoning по умолчанию для новых пользователей | допустимы `low`, `medium`, `high`, `xhigh`; по умолчанию `medium` |
| `CODEX_MODEL_OPTIONS` | optional override для списка моделей в Telegram UI | если не задан, используется встроенный curated list; если задан, полностью заменяет его |
| `CODEX_DEFAULT_LAUNCH_MODE` | режим по умолчанию для новых проектов | `sandbox` или `full_access` |
| `CODEX_SKIP_GIT_REPO_CHECK` | добавляет `--skip-git-repo-check` | по умолчанию `true` |
| `CODEX_TIMEOUT_SECONDS` | таймаут одного запуска Codex | должен быть больше `0` |
| `CODEX_ENABLE_IMAGES` | разрешает обработку фотографий через `--image` | по умолчанию `false` |

## 🖥️ Запуск в фоне

Бот рассчитан на один долгоживущий процесс, который получает обновления Telegram через `polling` и внутри себя запускает `codex exec`. Для первой проверки достаточно запуска в текущей консоли, но для постоянной работы удобнее использовать `tmux` или `systemd`.

Важно:

- на один bot token должен быть запущен только один `polling`-процесс
- если поднять второй экземпляр одновременно, Telegram вернёт `409 Conflict`
- если перезапустить процесс во время активного запроса, этот запрос не переживёт `restart`

### 🪟 Запуск через `tmux`

`tmux` удобен для личного сервера, VPS или локальной машины, когда нужен простой способ держать процесс запущенным и быстро подключаться к нему обратно.

Пример запуска:

```bash
cd /opt/codex-telegram-bot
source .venv/bin/activate
tmux new -s codex-bot
.venv/bin/python -m codex_telegram_bot.main
```

Полезные команды:

```bash
# запустить в фоне без входа в интерактивную сессию
tmux new -d -s codex-bot 'cd /opt/codex-telegram-bot && source .venv/bin/activate && .venv/bin/python -m codex_telegram_bot.main'

# подключиться к уже запущенной сессии
tmux attach -t codex-bot

# посмотреть список сессий
tmux ls

# остановить бота и удалить сессию
tmux kill-session -t codex-bot
```

Перед запуском проверь:

- `.env` заполнен корректно
- `codex` доступен в `PATH`
- другой экземпляр бота уже не запущен

### 🔧 Запуск через `systemd`

Для постоянного серверного запуска `systemd` обычно удобнее: он даёт автоматический перезапуск, автозапуск после `reboot` и стандартный доступ к логам через `journalctl`.

Пример unit-файла:

```ini
[Unit]
Description=Codex Telegram Bot
After=network.target

[Service]
Type=simple
User=codexbot
WorkingDirectory=/opt/codex-telegram-bot
EnvironmentFile=/opt/codex-telegram-bot/.env
ExecStart=/opt/codex-telegram-bot/.venv/bin/python -m codex_telegram_bot.main
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

Что важно:

- лучше запускать сервис от отдельного пользователя, а не от `root`
- `EnvironmentFile` должен указывать на рабочий `.env`
- `ExecStart` должен использовать Python из `.venv`
- пути в unit-файле нужно адаптировать под ваш сервер

Базовые команды:

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now codex-telegram-bot
sudo systemctl status codex-telegram-bot
sudo systemctl restart codex-telegram-bot
sudo journalctl -u codex-telegram-bot -f
```

## 🧪 Логи, SQLite и тесты

### 🗃️ Логи и SQLite

- логирование настроено через `structlog` и выводится в `JSON`
- SQLite работает в режиме `WAL`
- `project_sessions` хранит `thread_id`, `last_status`, `last_error`, `model_id`, `reasoning_effort` и время обновления
- `project_preferences` хранит `launch_mode` отдельно для каждого пользователя и проекта
- `user_preferences` хранит выбранные пользователем `model_id` и `reasoning_effort`
- `audit_log` хранит журнал аудита, в который события только добавляются, если `ENABLE_AUDIT_LOG=true`

### 🏗️ Коротко об архитектуре

- `main.py` запускает приложение, логирование, хранилище, предварительные проверки и `polling`
- `bot.py` собирает Telegram application и регистрирует обработчики
- `handlers` и `flows` разделяют навигацию и выполнение запросов
- `SessionStore` отвечает за SQLite-сессии, журнал аудита и настройки проекта
- `CodexRunner` адаптирует Codex CLI и нормализует поток событий
- `ProjectService`, `ObservabilityService` и `VoiceTranscriber` отвечают за маршрутизацию по рабочей области, журналирование и расшифровку
- `telegram/ui` и `telegram_formatting.py` отвечают за клавиатуры, тексты, `HTML`-рендеринг и разбиение длинных ответов

## 🚧 Текущие ограничения

- бот рассчитан на один долгоживущий `polling`-процесс на один bot token
- отдельной очереди задач и отдельного слоя исполнителей нет
- активный запрос не переживает `restart` процесса
- ограничитель частоты хранится в памяти процесса и сбрасывается при рестарте
- список проектов ограничен директориями первого уровня внутри `APPROVED_DIRECTORY`
- поддержка фотографий зависит от текущей поддержки `--image` в Codex CLI
- расшифровка голосовых сообщений ограничена внешними сервисами `openai` и `openai_compatible`
